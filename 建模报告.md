# 智慧城市共享停车位资源调配 - 建模报告

## 一、问题背景与数据概览

### 1.1 题目背景
本题研究智慧城市背景下的共享停车位资源调配问题。在共享经济模式下，私家车位主可在闲置时段将车位释放给其他用户使用，平台需根据用户需求和车位可用性进行最优调度，最大化资源利用率，同时控制用户步行距离。

### 1.2 数据集概览

| 数据集 | 规模 | 主要字段 |
|--------|------|----------|
| spots.csv | 300 个车位 | spot_id, x, y（坐标） |
| slots.csv | 477 条记录 | spot_id, start, end（可用时段） |
| users.csv | 500 个用户 | user_id, dest_x, dest_y, max_walk, arrival, departure |

**数据特征**：
- 空间范围：[0, 100] × [0, 100]
- 时间范围：8:00 - 22:00
- 车位时段分布：123 个单时段、177 个双时段
- 用户最大步行距离：5-25m，平均 15.33m
- 用户停车时长：均值 1.76 小时

### 1.3 子问题划分

| 问题 | 描述 | 求解类型 |
|------|------|----------|
| Q1 | 静态分配（无时间约束） | 整数规划 |
| Q2 | 时间约束下的分配 | 整数规划 + 冲突约束 |
| Q3 | 在线调度算法 | 在线算法 / 贪心 |
| Q4 | 供需分析与运营建议 | 数据分析 |

## 二、问题1：静态分配（无时间约束）

### 2.1 问题描述
忽略时间因素，将 500 个用户分配给 300 个车位，要求：
1. 每个用户最多分配一个车位
2. 每个车位最多服务一个用户
3. 用户与车位距离不超过用户最大步行距离

### 2.2 数学模型

**符号定义**：
- $U$：用户集合，$|U| = 500$
- $S$：车位集合，$|S| = 300$
- $d_{ji}$：用户 $j$ 到车位 $i$ 的欧氏距离
- $w_j$：用户 $j$ 的最大可接受步行距离
- $E = \{(j, i) : d_{ji} \leq w_j\}$：可行匹配边集

**决策变量**：
$$
x_{ji} \in \{0, 1\}, \quad \forall (j, i) \in E
$$

其中 $x_{ji} = 1$ 表示用户 $j$ 被分配到车位 $i$。

**两阶段优化**：

**阶段 1 - 最大基数匹配（Maximum Cardinality Matching）：**
$$
\begin{aligned}
\max \quad & \sum_{(j,i) \in E} x_{ji} \\
\text{s.t.} \quad & \sum_{i: (j,i) \in E} x_{ji} \leq 1, \quad \forall j \in U \\
& \sum_{j: (j,i) \in E} x_{ji} \leq 1, \quad \forall i \in S \\
& x_{ji} \in \{0, 1\}
\end{aligned}
$$

设阶段 1 最优值为 $N^*$。

**阶段 2 - 最小化总步行距离（Minimum Cost Bipartite Matching）：**
$$
\begin{aligned}
\min \quad & \sum_{(j,i) \in E} d_{ji} \cdot x_{ji} \\
\text{s.t.} \quad & \sum_{i: (j,i) \in E} x_{ji} \leq 1, \quad \forall j \in U \\
& \sum_{j: (j,i) \in E} x_{ji} \leq 1, \quad \forall i \in S \\
& \sum_{(j,i) \in E} x_{ji} = N^* \\
& x_{ji} \in \{0, 1\}
\end{aligned}
$$

### 2.3 数学方法

| 方法 | 应用 |
|------|------|
| **二分图匹配（Bipartite Matching）** | 用户-车位的二部图建模 |
| **最大基数匹配（Maximum Cardinality Bipartite Matching）** | 阶段1求最大可服务用户数 |
| **最小费用二分图匹配（Minimum Cost Bipartite Matching）** | 阶段2在最优用户数下最小化总距离 |
| **整数线性规划（ILP）** | 通过二元变量建模 |
| **字典序优化（Lexicographic Optimization）** | 主目标→次目标的两阶段求解策略 |

### 2.4 求解技术
- **求解器**：CBC（Coin-or Branch and Cut，开源 MILP 求解器）
- **建模框架**：PuLP（Python LP Modeler）
- **变量规模**：10,522 个二元变量
- **求解时间**：约 5.7 秒

> **理论说明**：二分图匹配问题可在多项式时间内求解（如匈牙利算法 $O(n^3)$、Hopcroft-Karp $O(E\sqrt{V})$）。本实现采用 ILP 是为了与 Q2 保持模型一致性，CBC 求解器能在小规模下快速给出最优解。

### 2.5 求解结果

| 指标 | 数值 |
|------|------|
| 服务用户数 | 300 |
| 平均步行距离 | 2.82 m |
| 车位利用率 | 100% |

## 三、问题2：时间约束下的分配

### 3.1 问题描述
在 Q1 基础上引入时间维度：
1. 车位有特定可用时段（如车位1：10:00-12:00 和 16:00-18:00）
2. 用户有到达和离开时间（如用户1：16:30-18:30）
3. **关键约束**：用户的整个停车时段必须完全包含在车位的某一连续可用时段内
4. **共享性**：同一车位在不同时间可服务多个互不重叠的用户

### 3.2 数学模型

**符号定义**：
- $T_i = \{[s_i^{(k)}, e_i^{(k)}]\}$：车位 $i$ 的可用时段集
- $[a_j, d_j]$：用户 $j$ 的停车时段
- **时间兼容**：$T(j, i) = 1 \Leftrightarrow \exists k: s_i^{(k)} \leq a_j \wedge d_j \leq e_i^{(k)}$
- **可行边集**：$E^T = \{(j, i) : (j,i) \in E \wedge T(j, i) = 1\}$

**车位内冲突**：
对每个车位 $i$，定义冲突对集合：
$$
C_i = \{(j_1, j_2) : j_1, j_2 \in U_i, j_1 \neq j_2, [a_{j_1}, d_{j_1}] \cap [a_{j_2}, d_{j_2}] \neq \emptyset\}
$$
其中 $U_i$ 是车位 $i$ 兼容的用户集合。两个时间区间重叠的判定：
$$
[a_1, d_1] \cap [a_2, d_2] \neq \emptyset \Leftrightarrow a_1 < d_2 \wedge a_2 < d_1
$$

**优化模型**：

**阶段 1（最大化服务用户）：**
$$
\begin{aligned}
\max \quad & \sum_{(j,i) \in E^T} x_{ji} \\
\text{s.t.} \quad & \sum_{i: (j,i) \in E^T} x_{ji} \leq 1, \quad \forall j \in U \\
& x_{j_1, i} + x_{j_2, i} \leq 1, \quad \forall i, \forall (j_1, j_2) \in C_i \\
& x_{ji} \in \{0, 1\}
\end{aligned}
$$

**阶段 2（最小化总步行距离，约束最大匹配数 $N^*$）：**
$$
\min \sum_{(j,i) \in E^T} d_{ji} \cdot x_{ji}
$$
约束同阶段 1，并增加 $\sum x_{ji} = N^*$。

### 3.3 数学方法

| 方法 | 应用 |
|------|------|
| **混合整数线性规划（MILP）** | 整体建模 |
| **区间图（Interval Graph）/ 互斥约束** | 同一车位不同用户的时间冲突建模 |
| **分组装箱思想（Conflict Graph Coloring）** | 每个车位上的用户构成区间冲突图 |
| **集合覆盖（Set Cover）扩展** | 用户对车位时段的覆盖关系 |
| **时间窗包含约束（Time Window Containment）** | 用户时段必须完全嵌入车位时段 |

### 3.4 实现要点
- 时间冲突约束以 pairwise 形式表达，每对冲突用户加一个约束
- 时间兼容性预处理过滤了 92.6% 的边（10522 → 776）
- 冲突约束数：1,135 对

### 3.5 求解结果

| 指标 | 数值 |
|------|------|
| 服务用户数 | 220 |
| 平均步行距离 | 9.26 m |
| 车位利用率 | 73.33% |
| 求解时间 | 约 3.0 秒 |

## 四、问题3：在线调度算法

### 4.1 问题描述
用户按到达时间顺序到达，平台必须立即对每个用户做出"接受并分配"或"拒绝"的决策，决策不可撤销。设计在线算法并分析其与离线最优解（Q2）的差距。

### 4.2 算法设计

**Greedy-Nearest 策略**：
```
Input: 按到达时间排序的用户序列
Output: 分配方案

for each user j (按到达时间)
    候选 ← {i ∈ S : 距离兼容(j,i) ∧ 时间兼容(j,i) ∧ 车位i在[a_j,d_j]空闲}
    if 候选非空:
        分配最近车位 i* = argmin_{i∈候选} d_{ji}
        更新车位i*的占用列表 += [a_j, d_j]
    else:
        拒绝用户j
```

### 4.3 数学方法

| 方法 | 应用 |
|------|------|
| **在线算法（Online Algorithm）** | 即时决策框架 |
| **贪心算法（Greedy Algorithm）** | Greedy-Nearest 策略 |
| **竞争分析（Competitive Analysis）** | 与离线最优 Q2 对比 |
| **在线二分图匹配（Online Bipartite Matching）** | 经典理论框架 |
| **Karp-Vazirani-Vazirani (KVV) 理论参考** | 在线匹配下界 1-1/e≈0.632 |

### 4.4 相对离线最优性能比

**定义**：经验性能比 $\rho = \frac{\text{ALG}}{\text{OPT}}$，其中 ALG 是在线算法服务的用户数，OPT 是离线最优解（Q2 结果）。该比值是**在当前数据集上的经验值**，而非理论竞争比。

**实测结果**：
$$
\rho = \frac{200}{220} \approx 0.909
$$

> ⚠️ **术语说明**：该值反映的是在线算法在当前数据集上相对于离线最优的表现，是"经验性能比"而非严格的"竞争比"（competitive ratio）。真正意义上的竞争比需要在所有可能实例上的最坏情况保证，需要理论证明。

### 4.5 求解结果

| 指标 | 数值 |
|------|------|
| 服务用户数 | 200 |
| 平均步行距离 | 9.47 m |
| 车位利用率 | 66.67% |
| 求解时间 | 约 1.4 秒 |
| 经验性能比（相对离线最优） | 90.9% |

## 五、问题4：供需分析与运营建议

### 5.1 宏观时段供需分析

**模型**：
- 时间离散化：将 [8, 22] 划分为 0.5 小时步长（28 个时间点）
- **供给** $S(t)$：时刻 $t$ 处可用的车位数 = $|\{i : \exists [s,e] \in T_i, s \leq t < e\}|$（半开区间）
- **需求** $D(t)$：时刻 $t$ 处活跃的用户数 = $|\{j : a_j \leq t < d_j\}|$（半开区间）
- **压力比** $P(t) = D(t) / S(t)$

> **限制说明**：此处压力比是**宏观时段压力比**，仅反映某时刻活跃用户与可用车位的数量比，不涉及空间可达性（步行距离）和时段完全覆盖约束。更精准的"真实可服务压力"需要空间-时间联合分析。

### 5.2 关键发现

| 时段 | 压力比 | 状态 |
|------|--------|------|
| 8:00 | 0.32 | 供过于求 |
| 13:00 | **3.04** | 最紧张 |
| 16:00-18:00 | 1.5-2.0 | 较紧张 |
| 19:00 后 | < 0.5 | 宽松 |

### 5.3 数学方法

| 方法 | 应用 |
|------|------|
| **时间序列分析** | 离散时间点的供需统计 |
| **核密度估计/直方图** | 用户最大步行容忍度分布 |
| **空间聚类分析** | 用户目的地热点识别 |

### 5.4 空间-时间压力热力图

为弥补宏观压力比的空间维缺失，新增了**网格化空间-时间压力分析**：

**方法**：
- 将 $[0,100] \times [0,100]$ 空间划分为 $10 \times 10$ 的网格
- 每个网格中心计算到所有车位的欧氏距离
- 统计每个网格在每个时刻：
  - **需求** $D(g,t)$：目的地在该网格内且在时刻 $t$ 活跃的用户数
  - **可达供给** $S_{\text{reach}}(g,t)$：从网格中心在 $\leq 14.9\text{m}$（用户步行距离中位数）内且时刻 $t$ 可用的车位数
  - **压力比** $\rho(g,t) = D(g,t) / S_{\text{reach}}(g,t)$

**关键发现**：

| 时刻 | 有需求网格 | 有供给网格 | 高压(>2) | 无供给 |
|------|-----------|-----------|----------|--------|
| 8:00 | 2 | 18 | 0 | 2 |
| 10:00 | 53 | 100 | 0 | 0 |
| 13:00 | 55 | 82 | **2** | **11** |
| 14:00 | 58 | 100 | 0 | 0 |
| 19:00 | 52 | 92 | 0 | 5 |
| 20:00 | 45 | 74 | 0 | 12 |
| 21:00 | 23 | 65 | 0 | 10 |

**结论**：
1. **13:00 是最关键时段**：11 个网格完全没有可达供给（压力比 inf），2 个网格压力比 > 2
2. **空间非均匀性明显**：中心区域 (55,55) 需求最集中（累计 17 用户），但供给充分；边缘区域供给不足
3. **早晚时段供给萎缩**：20:00 后 12 个网格无供给，适合推广"夜间共享"策略

**输出图表**：
- `q4_heatmap.png` — 7 个关键时刻（8h/10h/12h/14h/16h/18h/20h）的空间压力热力图
- `q4_heatmap_summary.png` — 供需总览（时间序列 + 空间分布 + 箱线图）

### 5.5 运营建议

1. **峰值定价（Peak Pricing）**：12:00-14:00 时段提高费率，利用价格杠杆调节需求
2. **错时共享激励**：补贴鼓励单时段车位主开放更多时段（敏感度分析显示可提升 39%-126%）
3. **步行优化**：用户高密度区域增加车位供给，推送精准导航
4. **智能预约推荐**：基于历史数据预测，主动推荐替代车位或时段

## 七、模型评估与改进建议

### 7.1 建模思路评估

| 问题 | 评估结论 | 评语 |
|------|----------|------|
| **Q1 静态分配** | ✅ 正确 | 两阶段 MILP 等价于二分图最小费用最大匹配。阶段1最大基数 + 阶段2最小化距离 = 词典序双目标优化。可改用 Hungarian 法（线性分配）直接求解，但保留 MILP 与 Q2 统一更易维护。 |
| **Q2 时间约束** | ✅ 正确 | pairwise conflict 建模正确，1135 个冲突对在规模上足够高效。时间兼容性检查（完全包含）和区间重叠判定均正确。更优方案：用"每个时刻容量≤1"的团约束代替 pairwise 形式。 |
| **Q3 在线调度** | ✅ 合理但非最优 | Greedy-Nearest 是合理 baseline，可解释性强。"90.9%"是经验性能比，非理论竞争比（需最坏情况保证）。实际表现良好。 |
| **Q4 供需分析** | ⚠️ 需补充分析 | 宏观压力比有参考价值，但未考虑空间可达性。建议增加空间-时间热力图（网格化）。 |

### 7.2 代码审查发现的问题

#### 问题 1：`utils.check_time_conflict()` 公式错误（高优先级）

**文件**：`utils.py:75-90`

**问题**：计算用户间时间冲突的向量化公式使用了错误的维度操作。
```python
# 错误代码（已修复）：
starts = intervals[:, 0].reshape(-1, 1)
ends = intervals[:, 1].reshape(1, -1)
conflict = (starts < ends.T) & (starts.T < ends)
```
第一项 `starts < ends.T` 本意是比较 `arrival_i < departure_j`，但实际运算变成了 `arrival_i < departure_i`（自身比较）；同理第二项 `starts.T < ends` 变成了 `arrival_j < departure_j`。只要两个用户自身停车时长为正，几乎所有用户对都会被误判为冲突。

**影响**：当前数据中产生 91,919 个误报冲突对（正确数应为 32,831）。该函数虽未被 Q2/Q3 主流程调用（它们各自独立实现了时间冲突检测），但影响任何使用此工具函数的分析代码。

**修复**：已修正为
```python
a = users["arrival"].to_numpy()  # (n,)
d = users["departure"].to_numpy()  # (n,)
conflict = (a[:, None] < d[None, :]) & (a[None, :] < d[:, None])
```

#### 问题 2：`q4_analysis.py` 时间区间端点不一致（中优先级）

**文件**：`q4_analysis.py:42,49`

**问题**：使用双端闭区间 `s <= t <= e` 和 `a <= t <= d`，而 Q2/Q3 的主逻辑使用半开区间 `[start, end)` 约定。这导致在端点处重复计数。

**影响**：对半小时粒度的时间序列分析影响有限（~1个时间点偏差），但概念上不一致。

**修复**：已改为 `s <= t < e` 和 `a <= t < d`。

#### 问题 3：Q4 压力比 inf 未过滤（中优先级）

**文件**：`q4_analysis.py:72`

**问题**：`idxmax()` 在 `supply=0` 时可能选到 `inf` 压力比时段，应同时过滤 `supply > 0`。

**修复**：已将过滤条件改为 `(demand > 0) & (supply > 0)`。

#### 问题 4：Q3 `greedy_balance` 负载度量偏差（低优先级）

**文件**：`q3_online.py:108-123`

**问题**：用 `len(self.occupancy[spot_id])` 表示负载，实际是历史累计服务次数，不是当前负载或未来压力。可能错误惩罚过去服务多但当前空闲的车位。

**建议**：改用当前时刻的实际占用区间数，或停车时段重叠数。

### 7.5 改进方案详述

#### Min-Regret（最小后悔）在线策略

当前 Greedy-Nearest 只考虑当前用户到候选车位的距离，忽略了对未来请求的影响。

**改进思路**：对每个候选车位 $i$ 评估"机会损失" $C_{ij}$，即如果占用车位 $i$，未来哪些用户可能因此被阻塞。

**简化实现**：
```python
# 机会成本 = 与当前用户时间重叠且以i为唯一/少数候选的用户数
def opportunity_cost(spot_idx, user_j, all_users, eligible, intervals):
    score = 0
    for k, u_k in all_users.iterrows():
        if k == user_j: continue
        # 时间重叠
        if intervals[user_j][0] >= intervals[k][1] or intervals[k][0] >= intervals[user_j][1]:
            continue
        eligible_set = eligible_ids[k]
        # 候选越少、依赖此车位越强 → 机会成本越高
        if len(eligible_set) > 0:
            score += 1.0 / len(eligible_set) if spot_idx in eligible_set else 0
    return score
```

#### 空间-时间压力热力图（Q4 改进）

将空间划分为 $10 \times 10$ 网格，对每个网格 $g$、每个时刻 $t$ 计算：

$$\rho(g,t) = \frac{|U(g,t)|}{|S_{\text{reachable}}(g,t)|}$$

其中 $U(g,t)$ 是网格 $g$ 内目的地在 $t$ 时刻活跃的用户数，$S_{\text{reachable}}(g,t)$ 是网格 $g$ 在步行距离内可达且 $t$ 时刻可用的车位数。
